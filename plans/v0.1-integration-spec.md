# v0.1 Integration Spec ÔÇö Auto-Discovery, Adaptive Scheduling, Per-Type Sensors

> Builds on [`HealthHandoff/03-integration-spec.md`](../HealthHandoff/03-integration-spec.md). That doc gave us a fixed coordinator-per-tool design with hardcoded intervals. **This spec replaces the fixed parts with discovery + adaptive scheduling**, and pins down per-data-type sensor formatting (ECG, HR, notifications, medications, lab/vitals, workouts).
>
> **Status:** draft for review. Not yet implemented. The current code at `custom_components/health_auto_export/` is the connectivity stub only.

---

## 0. Design goals (the user's requirements)

1. Connect to an iPad/iPhone running the HAE server and **pull as much information as possible**.
2. Create **persistent Home Assistant sensors**, formatted appropriately per data type ÔÇö ECG results, heart rate, heart-rate notifications, medications, lab results, workouts, vitals.
3. **Auto-discover** what the server exposes ÔÇö don't hardcode the tool/metric set.
4. **Classify update timing per sensor** based on the data's own historical update cadence. A sensor whose source updates hourly does not need minute-level polling.
5. Allow **per-domain override** so the user can pin a sensor to a faster or slower cadence than the auto-classification picked.

## 2. Discovery flow

### 2.1 Tool discovery (at integration setup + nightly refresh)

**Confirmed against a live HAE server (v0.0.1) during development.** The server advertises a working `listTools` RPC that returns the full tool catalog inline (each entry has `name`, `description`, and a JSON-Schema-style `inputSchema` for params and defaults). Standard MCP `tools/list` returns `-32601 Method not found` on this version, so we call legacy `listTools` first and only fall back to `tools/call`-style names if we ever encounter v1.0.0.

```python
async def discover_tools(client) -> list[ToolDescriptor]:
    # listTools is the authoritative source on v0.0.1.
    catalog = await client.list_tools()        # tries listTools, then tools/list
    if catalog is None:                        # both methods unsupported
        catalog = [ToolDescriptor(name=n) for n in KNOWN_TOOLS]
    # Confirm reachable: every advertised tool is exercised with its smallest legal args.
    confirmed = []
    for tool in catalog:
        result = await _probe_tool(client, tool)
        if result is not _UNKNOWN_TOOL:
            confirmed.append(tool)
    return confirmed
```

`_probe_tool` uses each tool's smallest legal arg shape (1-hour window for time-series, no metadata/routes for workouts, a single metric for `health_metrics`). Tool-unknown errors are signalled by **`-32602 Invalid params`** on this server, not `-32601` as we originally assumed; `-32601 Method not found` is reserved for unknown JSON-RPC *methods* (e.g. `tools/list`). The discovery loop treats both as "tool not available" and removes the entry. Transport failures abort discovery and surface via the status sensor.

**Tools confirmed advertised on the live device (8 + `listTools`):** `ecg`, `heart_notifications`, `workouts`, `health_metrics`, `medications`, `state_of_mind`, `symptoms`, `cycle_tracking`. The integration treats these as the authoritative ceiling ÔÇö there is **no** separate `sleep`, `mindfulness`, `audiogram`, `clinical_records`, or `lab_results` tool. Sleep / mindfulness / audio data comes through `health_metrics` (`sleep_analysis`, `mindful_minutes`, `environmental_audio_exposure`, etc.).

### 2.2 Metric discovery (`health_metrics`-specific)

`health_metrics` is the awkward one: there's no metric-list RPC. We pull a **30-day window with all `ALL_METRIC_IDS`** in one call (the server happily accepts a long comma-separated list and only returns what HealthKit actually has). The set of `name` values returned in `result.data.metrics[]` is the user's actual available-metric universe. Anything not returned is treated as "user doesn't have a device/source for this" and not turned into a sensor.

### 2.3 Empty-payload contract

The server signals "no data in this window" with `result.data = {}` (object, **not** `result.data = {<tool>: []}` or `null`). Discovery and per-tool coordinators must treat empty-object as success-with-no-items, not as a failure. Verified against `heart_notifications`, `state_of_mind`, `symptoms`, `cycle_tracking`, and `ecg` 1-hour probes on the live device.

### 2.4 Result of discovery

A `DiscoveryResult` recorded on the config entry:

```python
{
    "server_version": "v0.0.1",                    # from any successful response, when surfaceable
    "tools_available": ["ecg", "workouts", "heart_notifications", "health_metrics",
                        "medications", "state_of_mind", "symptoms", "cycle_tracking"],
    "metrics_available": ["heart_rate", "step_count", "blood_oxygen_saturation", ...],
    "medications_available": [                    # see ┬º3.5 for slug strategy
        {"slug": "example_rxnorm_123456", "display": "Example Medication 500mg"},
        ...
    ],
    "discovered_at": <iso>,
}
```

Sensors are created/removed from the entity registry on changes between runs. Removed sensors go `unavailable`, not deleted, so historical recorder data is preserved.

## 3. Sensor formatting per data type

Naming convention: `sensor.hae_<host_slug>_<datatype>_<field>`. Device-name suffix carries the host so multiple iPads coexist cleanly.

### 3.0 Storage tiers ÔÇö what each presentation layer reads

Every data type splits across **three storage tiers**, and dashboards (LCARS sickbay, default HA Lovelace, mobile app, third-party themes) consume whichever tier best fits the card they're rendering. The integration writes all three; the dashboard picks what it needs.

| Tier | Where it lives | Lifetime | Good for |
|---|---|---|---|
| **T1 ÔÇö Sensor state** | HA entity state + recorder | Forever (recorder retention) | Numeric trends, history graphs, gauges, simple value cards. Single scalar per entity. |
| **T2 ÔÇö Sensor attributes** | HA entity attributes (in-memory + state-stream snapshot) | Until next update | Compact structured data the card needs alongside the state (units, source device, classification, last N samples Ôëñ 60). Bounded payload ÔÇö never multi-MB. |
| **T3 ÔÇö File artifact** | `config/www/health_auto_import/<kind>/<id>.<ext>` served at `/local/...` (see ┬º3.10) | Bounded by per-kind retention | Rich medical-record visuals: ECG strips, GPX routes, hypnograms, adherence calendars, full raw JSON for export. Referenced by URL ÔÇö dashboards consume via `picture-entity`, `image`, `<img>`, GPX viewers, or `markdown` cards reading the file. |

**Rule of thumb for presentation builders:**
- Want a number on a tile? ÔåÆ T1 sensor state.
- Want to print "Sinus Rhythm, 78 BPM, 30 s @ 512 Hz" on a card? ÔåÆ T1 state + T2 attributes from the same sensor.
- Want to show an ECG trace, a workout map, a hypnogram, a prescription card with dosage history? ÔåÆ T3 file via the kind's `image.*` or `sensor.*_report_url` pointer entity (┬º3.10.2).

#### Why this matters for presentation parity

- **LCARS sickbay** can build the medical-record view by combining T1 + T3 ÔÇö sensors for the numeric panels, image entities for the strip/route/hypnogram, prescription JSON for the rx card. Zero coupling to integration internals.
- **Default HA Lovelace** users get a working "Health" dashboard from the included blueprint that uses stock `picture-entity` + `entities` + `markdown` cards, no HACS card dependencies.
- **Mobile / companion app** displays the same T1 + T2 data via standard entity rows; T3 URLs degrade gracefully (image renders in mobile too).
- **Third-party themes** (HomeKit-style, minimalist) consume whichever tier fits their language.

The integration's job is to produce T1, T2, and T3 faithfully; the presentation layer's job is to pick from the menu. Neither side has to know the other exists.

### 3.1 ECG (`tool: ecg`)
One ECG record set per HealthKit ECG taken on the watch. The integration tracks the **latest** record and counts. **Field names verified against a live server response.**

Real record shape (voltage stripped):
```json
{
  "start": "2025-01-15 08:17:01 -0700",
  "end":   "2025-01-15 08:17:31 -0700",
  "averageHeartRate": 78,
  "classification": "Sinus Rhythm",
  "source": "ECG",
  "samplingFrequency": 512,
  "numberOfVoltageMeasurements": 15360,
  "voltageMeasurements": [...]
}
```

| Sensor | Native value | Device class | State class | Source field |
|---|---|---|---|---|
| `ecg_last_classification` | string ÔÇö observed: `Sinus Rhythm`, `Atrial Fibrillation`, `Inconclusive`, `Unclassifiable` (HAE returns title-case strings, not camelCase as our earlier draft assumed) | ÔÇö | ÔÇö | `classification` |
| `ecg_last_average_bpm` | int | ÔÇö | `measurement` (unit `bpm`) | `averageHeartRate` |
| `ecg_last_taken_at` | datetime | `timestamp` | ÔÇö | `start` |
| `ecg_last_duration_s` | float | `duration` | `measurement` | computed: `end - start` (typically 30 s) |
| `ecg_last_sampling_frequency_hz` | int | `frequency` | `measurement` | `samplingFrequency` |
| `ecg_last_voltage_count` | int | ÔÇö | `measurement` | `numberOfVoltageMeasurements` |
| `ecg_count_30d` | int | ÔÇö | `total` | rolling 30-day window |

Voltage samples are **never** the state. Gated by config option `include_ecg_voltage` (default OFF) ÔÇö when ON, the last record's `voltageMeasurements` array goes on the `last_classification` sensor's attribute, marked with `entity_registry_visible_default=False`.

**Default args:** `includeMetadata: false`, `includeRoutes: false`. The server-advertised default for `includeMetadata` is **true** (verified via `listTools`), which embeds large per-minute time-series arrays per metric ÔÇö we explicitly override to `false` because they multiply payload size and don't make good entity state.

Real record shape (top-level fields verified):
```json
{
  "name": "Outdoor Walk",
  "location": "Outdoor",
  "start": "2025-01-15 07:29:25 -0700",
  "end":   "2025-01-15 07:54:50 -0700",
  "duration": 1525.10,
  "activeEnergyBurned": {"qty": 72.16, "units": "kcal"},
  "temperature":        {"qty": 10.94, "units": "degC"},
```

### 3.4 Vitals & lifestyle (`tool: health_metrics`)
One sensor per **discovered** metric (see ┬º2.2). Format driven by the metric's `units` field in the response ÔÇö no hardcoded unit table.

| Pattern | Native value | Device class (HA) | State class |
|---|---|---|---|
| `metric_<id>_latest` | latest `qty` from data array | inferred from `units` | `measurement` for instantaneous, `total_increasing` for cumulative (step_count, active_energy, flights_climbed, apple_exercise_time) |
| `metric_<id>_today_total` | sum of today's points | same as above | `total_increasing` (cumulative metrics only) |

`units` ÔåÆ `device_class` mapping:

```
bpm                   ÔåÆ none, state_class=measurement
count, count/min      ÔåÆ none
%                     ÔåÆ none, unit %
degC, degF            ÔåÆ temperature
mmHg                  ÔåÆ atmospheric_pressure (closest fit; blood_pressure handled below)
mg/dL                 ÔåÆ none, custom unit
kcal                  ÔåÆ energy
m, km, mi, ft, yd     ÔåÆ distance
m/s, km/hr, mph       ÔåÆ speed
ms                    ÔåÆ duration
hr, min, s            ÔåÆ duration
L, mL                 ÔåÆ volume
g, mg, ┬Ág, kg, lb     ÔåÆ weight (mass)
mcV                   ÔåÆ voltage (ECG only ÔÇö not used as a metric sensor)
```

**Blood pressure** is a composite metric **inside `health_metrics`**, confirmed live: each data point is `{date, source, systolic, diastolic}` (no `qty`/`units` keys). Split into two sensors `metric_blood_pressure_systolic_latest` and `metric_blood_pressure_diastolic_latest` (both `unit_of_measurement: mmHg`, `state_class: measurement`), plus a string composite `metric_blood_pressure_latest` with state `"120/80"`.

```json
  "inBed":        8.38,
  "sleepStart":   "2025-01-15 00:35:59 -0700",
  "sleepEnd":     "2025-01-15 08:58:59 -0700",
```

### 3.7 Server status, reachability & notifications

The TCP server lives on a phone or tablet that will go to sleep. **Surfacing that clearly is a first-class feature**, not a diagnostic afterthought ÔÇö without it the user's "vitals offline" experience is a silent stale graph.

#### Entities

| Entity | Type | Purpose |
|---|---|---|
| `binary_sensor.health_auto_import_<host>_reachable` | `connectivity` | ON when the last 30 s TCP probe + JSON-RPC ping succeeded. Primary trigger for offline notifications. |
| `sensor.health_auto_import_<host>_status` | enum sensor | Coarse human-readable state. One of: `online`, `degraded`, `unreachable`, `unauthorized`. See state machine below. |
| `sensor.health_auto_import_<host>_last_seen` | `timestamp` | Wall-clock of the last successful response from the server (any tool). Drives "offline for N minutes" templates without timer helpers. |
| `sensor.health_auto_import_<host>_last_error` | text | Most recent failure message (transport message, JSON-RPC `error.message`, or `null`). Diagnostic only ÔÇö `entity_category: diagnostic`. |
| `sensor.health_auto_import_<host>_consecutive_failures` | int, `measurement` | Counter of consecutive failed coordinator updates across all tools. Resets to 0 on any success. |
| `sensor.health_auto_import_<host>_discovered_tools_count` / `_metrics_count` | int | Output of the discovery coordinator. Diagnostic. |

`<host>` is the IP or hostname slug so multi-iPad setups don't collide.

#### Status state machine (`sensor.health_auto_import_<host>_status`)

| State | Enters when | Exits when |
|---|---|---|
| `online` | Reachability probe succeeds **and** every active tool coordinator's last update succeeded | Any of the conditions below |
| `degraded` | Reachability probe succeeds **but** one or more tool coordinators failed (e.g. `medications` returns `-32601`) | All tool coordinators succeed ÔåÆ `online`; reachability fails ÔåÆ `unreachable` |
| `unreachable` | TCP connect failed **or** read timed out on the reachability coordinator | Next successful probe ÔåÆ `online`/`degraded` |
| `unauthorized` | Reserved: JSON-RPC error code mapped to auth (HAE v0.0.1 does not authenticate today, but v1.0.0 might). Not entered in v0.2. | ÔÇö |

The status sensor's value is computed on every coordinator update ÔÇö no separate poll. Attributes include `last_change_at`, `unreachable_since`, and the contributing tool error map.

### 3.9 Watermarks, backfill, and gap recovery

The TCP server lives on a device that sleeps, gets backgrounded, leaves the LAN, or has HAE killed by iOS. The integration must never (a) miss records that arrived while it was disconnected, (b) re-emit records it already ingested, or (c) fire a year-long query on the first poll after a long outage. **High-water-mark + bounded catch-up** solves all three.

#### 3.9.1 Per-tool state (persisted on the config entry)

Stored in `entry.data["watermarks"][<tool>]`:

```python
{
    "initial_crawl_done":  True,
    "initial_crawl_at":    "2025-01-15T15:00:00-07:00",
    "watermark":           "2025-01-13T09:14:33-07:00",   # newest record.end ingested
    "last_success_at":     "2025-01-15T15:30:00-07:00",   # last clean poll
    "seen_keys":           [<bounded LRU of Ôëñ 500 dedup keys>],
}
```

Persistence is synchronous on every successful poll (`async_update_entry`) so a restart mid-cycle never re-replays or skips.

#### 3.9.2 First load ÔÇö backward crawl (sparse tools only)

Applies to: `heart_notifications`, `ecg`, `medications`, and the opt-in tools (`state_of_mind`, `symptoms`, `cycle_tracking`) when enabled.

```python
async def initial_crawl(client, tool, *, step=timedelta(days=30),
                        floor=timedelta(days=365*2)):
    end = utcnow()
    horizon = end - floor
    while end > horizon:
        start = max(end - step, horizon)
        payload = await client.call_tool(tool, {"start": fmt(start), "end": fmt(end)})
        records = client.extract_payload(payload, tool)
        if records:
            watermark = max(r["end"] for r in records)
            return watermark, records
        end = start
    return install_date, []        # nothing in 2 years ÔåÆ wait for live events
```

Rules:
- Step size 30 days; floor `install_date ÔêÆ 2 years` (configurable, advanced option).
- Runs once per tool, marked `initial_crawl_done = True` on completion. Surface a progress notification via `persistent_notification` while crawling ("Discovering historical heart_notificationsÔÇª scanning 2024-Q3").
- Crawl is **resumable**: persist `watermark` after each window so a HA restart picks up where it left off.
- Dense tools (`workouts`, `health_metrics`) **skip the crawl** ÔÇö set `watermark = now ÔêÆ 7 days` so first-load dashboards aren't empty, and let normal polling fill forward.

#### 3.9.3 Polling ÔÇö forward-only with bounded catch-up

```python
async def poll(client, tool, state):
    now = utcnow()
    window_start = state["watermark"] - OVERLAP[tool]
    window_start = max(window_start, now - MAX_CATCH_UP[tool])     # cap!
    payload = await client.call_tool(tool, {"start": fmt(window_start), "end": fmt(now)})
    records = client.extract_payload(payload, tool)
    new = [r for r in records if dedup_key(tool, r) not in state["seen_keys"]]
    for r in new:
        emit(r)
        state["seen_keys"].append(dedup_key(tool, r))
        state["watermark"] = max(state["watermark"], r["end"])
    state["last_success_at"] = now
    _trim_lru(state["seen_keys"], 500)
    persist(state)
```

**Two knobs that matter:**

| Constant | Default | Why |
|---|---|---|
| `OVERLAP[<tool>]` | 5 min sparse / 30 min `health_metrics` | HealthKit backfills from a watch that just resynced; overlap catches late arrivals. The dedup LRU prevents re-emission. |
| `MAX_CATCH_UP[<tool>]` | 14 d sparse / 7 d `health_metrics` / 30 d `workouts` | Caps the catch-up window so a multi-week outage doesn't fire a mega-query that times out or runs the iPad battery flat. |

When `now - watermark > MAX_CATCH_UP`, ingest what we can, then **raise a Repairs issue** (┬º3.7) with a one-click action that invokes `health_auto_import.backfill_tool` to recover the gap on-demand.

#### 3.9.4 Dedup keys per tool

A record's identity is the tuple that uniquely identifies it; overlap windows would otherwise re-emit recent records every poll.

| Tool | Dedup key |
|---|---|
| `ecg` | `start` |
| `heart_notifications` | `(start, type)` |
| `workouts` | `(start, name)` |
| `health_metrics` (per metric) | `(metric_name, date, source)` |
| `medications` | `(rxnorm_code or displayText, scheduledDate)` |
| `state_of_mind` | `(date, kind)` |
| `symptoms` | `(date, name)` |
| `cycle_tracking` | `(date, kind)` |

LRU is bounded at 500 entries per tool. Anything older than the watermark can't reappear (overlap is far shorter than the LRU horizon), so eviction is safe.

#### 3.9.5 Behaviour at boundary conditions

| Scenario | Behaviour |
|---|---|
| Server down 5 min | Next poll asks `watermark - OVERLAP ÔåÆ now`. Catches the gap. No special handling. |
| Server down 2 weeks | Sparse tools: full catch-up window fetched, no issue raised. `health_metrics`: capped at 7 days, Repairs issue raised offering `backfill_tool`. |
| HA restart mid-poll | `last_success_at` not updated; watermark not advanced. Next poll re-fetches the in-flight window. Dedup LRU prevents duplicates. |
| iPad HealthKit reauth (history truncated) | New polls return fewer records; watermark stays where it was; sensors go stale until new events. No false replays. |
| User uninstalls and reinstalls | New config entry ÔåÆ fresh state ÔåÆ initial crawl runs again. Old recorder history preserved (entity unique_ids stable). |
| HAE app upgraded (new tool appears) | Discovery picks it up; new tool runs its own initial crawl; existing tools unaffected. |

#### 3.9.6 On-demand services

Live in `services.yaml`:

| Service | Purpose |
|---|---|
| `health_auto_import.backfill_tool` | One-shot. Args: `tool`, optional `start` / `end` (default: `last_success_at ÔåÆ now`). Walks back in 30-day windows, ingests via the normal dedup path, advances watermark. Surfaces progress via `persistent_notification`. |
| `health_auto_import.fetch_window` | One-shot, **non-mutating**. Args: `tool`, `start`, `end`. Returns the raw payload as a service response for scripts/templates. Does not touch watermark, dedup, or entity state. |
| `health_auto_import.reset_watermark` | Args: `tool`, optional `to` (default: `install_date`). Resets state and triggers an initial crawl on next coordinator tick. Used as escape hatch after device migrations. |

All three are `entity_category: diagnostic` from a UI standpoint ÔÇö advanced users only; not surfaced in the default Lovelace card.

#### 3.10.4 Renderer / coordinator separation (architectural rule)

- **Coordinators never touch report files.** They emit "new record ingested" events on the event bus.

- **Reports are regeneratable.** If a user deletes `www/health_auto_import/` or upgrades the integration with a new render version, calling `health_auto_import.rerender_reports` walks the dedup LRU and rebuilds every artifact from cached raw data.

#### 3.10.5 Storage hygiene

- Default retention: **30 most-recent records per kind**. Older files GC'd nightly. Configurable via options (`keep_last_ecgs`, `keep_last_workouts`, etc.) with floors (ÔëÑ3) so a misconfigured `0` can't blank the dashboard.
- `medications/<slug>_adherence_30d.svg` is overwritten in place daily ÔÇö only ever one file per drug.
- Total expected footprint with defaults: < 5 MB across all kinds.

## 4. Adaptive scheduler

The point: don't poll `weight_body_mass` every minute when the user weighs themselves weekly, and don't poll `heart_rate` every hour when the watch writes new samples every 5 minutes.

### 4.1 Observation model

For each `(tool, metric)` the coordinator records, on every successful fetch, the **timestamp of the newest data point** the server returned. Sliding 14-day buffer of these "newest data point" timestamps per sensor (in-memory, persisted to `.storage/health_auto_export_<entry_id>.json` so restarts don't lose history).

> **Interaction with watermarks (┬º3.9):** the adaptive scheduler decides *how often* a coordinator polls; the watermark decides *what window* each poll requests. They are independent. A long poll interval simply means each request covers a longer catch-up window ÔÇö still bounded by `MAX_CATCH_UP`. The scheduler must never advance the watermark on its own; only successful ingest does that.

### 4.2 Per-sensor cadence recommendation

```python
def recommended_interval(timestamps: list[datetime]) -> timedelta:
    if len(timestamps) < 3:
        return DEFAULT_BY_TOOL[tool]           # fall back to const.py default
    gaps = sorted(t2 - t1 for t1, t2 in pairwise(timestamps))
    median = gaps[len(gaps) // 2]
    # Sample twice per natural update gap. Floor and ceiling tame outliers.
    return clamp(median / 2, ADAPTIVE_MIN_INTERVAL_S, ADAPTIVE_MAX_INTERVAL_S)
```

Sampling at half the median gap means we catch a new point within one polling cycle of it landing, without hammering the server. `ADAPTIVE_MIN = 30 s`, `ADAPTIVE_MAX = 1 h`.

### 4.3 Coordinator cadence vs. sensor cadence

Coordinators pull **per tool**, not per sensor. So:

- For `health_metrics`, the coordinator's interval = `min(recommended_interval for all metrics in this tool, clamped)`. Cheap because one call returns all metrics.
- For `ecg` / `workouts` / `heart_notifications` / `medications`, the coordinator's interval = that tool's single recommendation.
- The reachability coordinator stays fixed at 30 s.

### 4.4 Re-evaluation

The discovery coordinator re-runs nightly and recomputes recommendations. Any coordinator whose recommended interval drifts by > 20 % gets `update_interval` rewritten on the spot ÔÇö no reload needed.

### 4.5 User overrides

Options flow exposes a per-tool override (and, optionally, per-metric override for `health_metrics`):

| Override | Effect |
|---|---|
| `auto` (default) | Adaptive scheduler picks the interval. |
| `<N> seconds/minutes` | Hard pin. Bypasses the scheduler floor/ceiling. |
| `disabled` | Coordinator does not poll. Sensors go `unavailable`. |

## 5. Config flow

The first step the user sees is a **picker** ÔÇö auto-scan results plus an always-available "Enter manually" option. The user is never forced to type an IP, and never forced to wait for a scan.

### 5.1 Setup ÔÇö `async_step_user` (v0.2)

1. **Kick off a background LAN scan** as soon as the step is opened. The form renders immediately; results trickle in. Scan strategy:
   - **Zeroconf / mDNS first.** Subscribe to `_health-export._tcp.local.` (and `_http._tcp.local.` as a fallback) via `homeassistant.components.zeroconf.async_get_async_instance`. Each hit is JSON-RPC-probed with a 1-hour `health_metrics` call to confirm it's HAE and not some other service on `:9000`.

### 5.2 Zeroconf auto-trigger (passive discovery)

In `manifest.json`:

```json
"zeroconf": [
  { "type": "_health-export._tcp.local." }
]
```

When HA itself spots a matching service (even when the user hasn't opened any integration UI), it fires `async_step_zeroconf`. We pre-fill the picker with that host and skip straight to the confirm + discovery steps if the user accepts the discovery banner. Service type is **TO VERIFY** against a live HAE install ÔÇö if HAE doesn't advertise mDNS, this clause becomes a no-op and the user-initiated picker is the only path.

### 5.3 Options flow (post-setup reconfiguration)

- Re-run discovery on demand (`[ Rescan device ]` button).
- Per-tool / per-metric override table from ┬º4.5.
- `include_ecg_voltage` toggle.
- Adaptive scheduler min/max bounds (advanced, defaulted from `const.py`).

### 5.4 Already shipped in v0.1

Manual host+port single-step form with a TCP probe ÔÇö kept around as the fallback path inside the picker. The picker UI and scan logic are the v0.2 additions.

## 6. Error handling rules (verified against live v0.0.1 server)

| Condition | What flips | Logs |
|---|---|---|
| TCP connect fails | reachability OFF, **all** tool coordinators ÔåÆ `UpdateFailed` | WARN once per state change |
| TCP connect OK, read timeout | reachability OFF, that coordinator ÔåÆ `UpdateFailed` | WARN |
| JSON-RPC `error.code = -32601 Method not found` | RPC *method* (e.g. `tools/list`) not supported on this server version. Caller should fall back (`tools/list` ÔåÆ `listTools`); if no fallback exists, surface as `UpdateFailed`. | INFO |
| JSON-RPC `error.code = -32602 Invalid params` with "Tool not found" / unknown-tool message | The named *tool* doesn't exist on this device. Drop from discovery; coordinator removed. **Not** an error condition for the user. | DEBUG |
| JSON-RPC `error.code = -32602` with other messages | Bad arguments ÔÇö our bug. Coordinator ÔåÆ `UpdateFailed` with the RPC message. | ERROR |
| Got `result.data == {}` (empty object, not array, not null) | reachability ON; coordinator data ÔåÆ empty list; "latest" sensors ÔåÆ `unknown`, counts ÔåÆ `0` | DEBUG |
| Got `result.data.<key>` populated | reachability ON; sensors update normally | DEBUG |

Per-call timeout: 30 s. HA's built-in `DataUpdateCoordinator` exponential backoff handles repeated failures. The integration distinguishes "tool doesn't exist" (silent removal) from "tool exists but server bug" (ERROR log) by inspecting the error message text on `-32602`.

## 7. Storage & recorder considerations

- ECG voltage arrays: **never** in entity state. Gated attribute only. Even gated, mark sensor `entity_registry_visible_default=False`.
- HRN HR/HRV time-series during an event: attribute on `hrn_last_at`, capped at the most recent 60 samples; full payload goes to a one-shot `health_auto_export.dump_last_hrn` service (v0.3).
- Workout route data: never fetched by default (`includeRoutes=false`). A `health_auto_export.fetch_workout_route` service in v0.3 can grab one on demand and write it to a file.

The integration must **not** put multi-MB blobs into the HA recorder.

## 9. Open questions for review

1. **Discovery cadence.** Nightly + on-setup. Should we also re-discover on every successful reachability OFFÔåÆON transition (user just woke the iPad after days)? *Lean: yes, but rate-limit to once per hour.*
2. **Metric override granularity.** Per-tool is easy. Per-metric inside `health_metrics` means the coordinator can't be a single object ÔÇö would need either multiple `health_metrics` coordinators (one per cadence group) or post-coordinator filtering. *Lean: cadence-group buckets ÔÇö auto-cluster metrics into 2ÔÇô4 cadence tiers and run one coordinator per tier.*
3. **Sleep entity shape.** ┬º3.4 has 7 sleep sensors. Tradeoff: lots of entities vs. one `sensor.sleep_last_summary` with the breakdown as attributes. *Lean: keep them split ÔÇö HA history graphs work better on numeric state than on attributes.*
4. **Medication slug stability.** ~~If the user renames a med in Apple Health, the slug changes.~~ **RESOLVED via live probe:** records include `codings[]` with RxNorm `system`/`code`. Use `rx_<rxnorm_code>` as the primary slug ÔÇö survives display-name renames. Fallback to slugified `displayText` only when no RxNorm coding is present. See ┬º3.5 for the slug builder.
5. **Blood pressure composite.** Worth the third synthetic entity, or just attributes on systolic? *Lean: keep the composite ÔÇö dashboards want to print "120/80" without templating.*
